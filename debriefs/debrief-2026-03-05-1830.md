## Learnings

### Tailwind v4 animation tokens must live inside @theme
**Context:** Added a `hero-sweep` CSS animation with `@keyframes` defined outside the `@theme` block, then referenced it with `animate-hero-sweep`. The animation never rendered.
**Learning:** In Tailwind v4, custom animation utilities (`--animate-*`) and their associated `@keyframes` must be declared inside the `@theme { }` block. Keyframes defined at the root CSS level won't be picked up by Tailwind's class generation. This is different from Tailwind v3 where you could define keyframes anywhere and reference them with arbitrary values.
**Why it matters:** Any time you add a custom animation to this project or any Tailwind v4 project.

### Don't overdesign treatments on top of video elements
**Context:** Built layered overlays on the hero video (gradient vignettes, grain texture, animated light sweep). The user couldn't see any of it and asked to remove it all.
**Learning:** Video elements are already visually complex and dynamic. Adding subtle CSS overlays on top of video is almost always invisible to users, especially at low opacity values. If the video needs treatment, do it to the video file itself (color grade, vignette baked in) rather than layering CSS on top. A simple color tint (`bg-cream/10`) is usually sufficient for blending the video edge into the page.
**Why it matters:** General principle for any hero video implementation. Save the time.

### Validate the scale of the problem before building complex UI patterns
**Context:** Built an entire subtype chip-tab system for the flooring carousel cards (with auto-cycling, AnimatePresence transitions, intersection observer). The user then realized there are only ~9-10 flooring types total, which is perfectly fine as flat cards in a carousel. The feature was reverted.
**Learning:** When a client proposes a UI pattern to solve a perceived scale problem ("it could be like a dozen boxes"), verify the actual data volume first. 9-10 items in a horizontal carousel is not a problem that needs solving. Ask "how many items will there actually be?" before building grouping/nesting UI.
**Why it matters:** This is a recurring pattern in client work. Clients anticipate complexity that doesn't materialize. A 2-minute data audit saves hours of UI work.

### Wix static image URLs work as external sources
**Context:** Needed the Ecomedes logo. Found it on their Wix-hosted logos page. Used the `static.wixstatic.com` URL directly as an `<img>` src.
**Learning:** Wix static asset URLs (`static.wixstatic.com/media/...`) are stable, publicly accessible, and work fine as external image sources. They support format parameters like `/v1/fill/w_288,h_288,al_c,q_85,enc_avif,quality_auto/`. Good for prototyping when you need a client's logo but don't have the actual asset file yet.
**Why it matters:** Quick way to pull in partner/client logos during prototyping without waiting for asset delivery.

### Firecrawl auth requires browser flow
**Context:** Installed firecrawl-cli globally but it wasn't authenticated. Had to run `firecrawl login --browser` which opens a browser window for OAuth.
**Learning:** Firecrawl CLI requires `firecrawl login --browser` for first-time auth. It opens a browser automatically. The `--browser` flag prevents it from prompting interactively, which is important when running from an agent. Auth persists across sessions after the first login.
**Why it matters:** Project-general. First time setting up firecrawl in any new environment.

## Skill & Automation Candidates

### vercel-preview-deploy
**What it would do:** Wrap the three-step Vercel preview deploy workflow (build check, `vercel`, `vercel alias`) into a single command. Includes the alias step automatically using a project-configured preview URL.
**Why it's worth building:** This exact sequence (build, deploy, alias to rfci-preview.vercel.app) was run 6+ times in this session. Each time it's three commands with a long deployment URL to copy-paste for the alias.
**Trigger:** "deploy" or "deploy preview" on any project with a configured preview alias.
**Complexity:** Low. Shell script or a simple skill that reads the alias from a config.

### light-dark-section-theme-swap
**What it would do:** Take a section component and swap it between light and dark theme variants, automatically converting all the color classes (white/XX to rfci-black/XX, bg-white/5 to bg-rfci-blue/5, etc.).
**Why it's worth building:** The StandardsSection dark-to-light conversion touched 30+ color classes across desktop tabs, mobile accordion, stat counters, and animated content. It's a systematic find-and-replace pattern that could be automated.
**Trigger:** "Make this section light/dark"
**Complexity:** Medium. The mapping isn't 1:1 in every case. Needs judgment about contrast ratios on different backgrounds.

## LinkedIn Content

### Post 1: The feature I built, shipped, and deleted in 20 minutes

I built a whole subtype tab system for a flooring carousel today. Chip toggles, auto-cycling animation, intersection observer to trigger the first demo cycle, AnimatePresence crossfades. The whole thing.

Shipped it to preview. Looked at it. Realized the client's concern was "what if there are too many cards" and the answer was... there are 9 cards. Nine. In a horizontal carousel with scroll arrows and a progress bar.

Nine cards is not a problem.

Reverted the whole thing. Back to flat cards. Took about 3 minutes to undo 20 minutes of work.

The instinct when a client says "we need to group these" is to start building grouping UI. The better instinct is to count how many items there actually are. If it's under 15 in a carousel, you don't have a grouping problem, you have a client imagining a grouping problem.

I keep relearning this. Validate the scale of the problem before you build the solution. A 2-minute data audit would have saved the whole detour.
